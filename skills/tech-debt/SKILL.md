# Tech Debt — створення і редагування

Скіл фіксує і оновлює технічний борг у репозиторії проєкту за усталеною
структурою OneB: реєстр-таблиця + окремий файл на кожен борг. Покриває два
сценарії:

1. **Створення** нового боргу
2. **Редагування** наявного боргу (метадані, опис, варіанти, рекомендація,
   критерії, статус «відкладено»/«відкрито»)

**Що цей скіл не робить:**

- **Виконання** (закриття) боргу — це `tech-debt-close`.
- **Видалення** боргу як неактуального — це теж `tech-debt-close`.
- **Проактивна поведінка** під час будь-якої задачі з кодом — це
  `tech-debt-proactive`.

Скіл сам не комітить і не пушить — це робить користувач.

## Коли активуватись

### Створення нового боргу

**Явні тригери — діяти одразу:**

- «запиши в техборг» / «додай в техборг» / «запиши тех борг» / «відкрий тех борг»
- «новий тех борг» / «новий tech debt»
- «record tech debt» / «log tech debt» / «open tech debt item» / «tech debt item»
- Згадка `tech_debt/` у контексті запису

**Неявні сигнали — спершу запропонувати:**

- «в майбутньому треба буде зробити...»
- «потім ми будемо робити...»
- «це поки заглушка»
- «треба буде колись поправити»
- «зараз пропустимо, але повернемось»
- «це не наш кейс зараз, але...»

При неявному сигналі: коротко запитати в користувача «Зафіксувати це як тех
борг у `tech_debt/`?» і чекати «так». Без явного підтвердження запис не
створювати.

### Редагування наявного боргу

**Явні тригери:**

- «онови тех борг NNN» / «зміни тех борг NNN»
- «зміни пріоритет тех боргу NNN на ...»
- «додай критерій у борг NNN» / «онови варіанти у борг NNN»
- «зміни рекомендацію у борг NNN» / «онови опис боргу NNN»
- «перенеси борг NNN у відкладені» / «зніми відкладеність з борга NNN»
- «edit tech debt NNN» / «update tech debt NNN»

## Структура каталогу

```
tech_debt/
├── index.md                       # реєстр-таблиця всіх відкритих боргів
└── items/
    ├── 001-kebab-case-slug.md
    ├── 002-another-slug.md
    └── ...
```

Іменування файлів: `NNN-kebab-case-slug.md`, де `NNN` — трицифровий номер.

Номери **не переприсвоюються** при видаленні/закритті: якщо закрили `005`,
наступний новий борг отримує номер `(макс + 1)`, а не `005`. Так посилання
з git-історії та з коду на `#005` залишаються однозначними.

## Реєстр (tech_debt/index.md)

Таблиця з колонками: `#`, `Назва (з посиланням)`, `Категорія`, `Пріоритет`,
`Статус`.

**Пріоритети** — три рівні з кольоровими маркерами:

- `🔴 високий` — блокує або спричиняє баги
- `🟡 середній` — впливає на DX/UX
- `🟢 низький` — приємно мати

**Статуси** — два значення:

- `відкрито` — готовий до взяття у роботу
- `🕒 відкладено` — чекає на умову активації (дата або реалізація іншого
  функціоналу); в item-файлі обов'язково вказано `Активувати після:`

Закриті/виконані/видалені борги у реєстрі не зберігаються — історія в git.

Точний шаблон реєстру — у `references/index-template.md`. Зчитати, коли
треба створити `index.md` з нуля або переконатись у форматі рядка.

## Структура item-файлу

Шість блоків у фіксованому порядку:

1. **Заголовок** — `# NNN — Коротка назва`
2. **Метадані** (буліст):
   - `Категорія:` — вільне поле (приклади: `Backend / Domain model`,
     `DX / Build`, `Observability / Operations`, `Frontend / State`,
     `Testing`, `Security`, `Performance`)
   - `Пріоритет:` — з емодзі (`🔴 високий` / `🟡 середній` / `🟢 низький`)
   - `Статус:` — `відкрито` або `🕒 відкладено`
   - `Активувати після:` — опційне поле; **обов'язкове, якщо статус
     `🕒 відкладено`**. Значення: конкретна дата (`2026-09-01`),
     реліз/мілстоун (`після релізу v2.0`), або умова з кодової бази
     (`після впровадження auth-модуля`).
   - `Дотичні файли:` — список markdown-посилань на конкретні файли
     (`[path/to/file.ts](path/to/file.ts)`) + коротке пояснення поряд
3. **`## Проблема`** — контекст: що саме не так, чому це болить, які наслідки
   видно зараз. Може містити кодові блоки, цитати з логів.
4. **`## Куди рухатись`** — варіанти рішення (`Варіант A`, `Варіант B`,
   `Варіант C` — мінімум один) з плюсами/мінусами, опційно блоки коду.
   У кінці — `Попередня рекомендація:` з обґрунтуванням.
5. **`## Критерії "зроблено"`** — чек-лист `- [ ]` з конкретними
   перевірюваними умовами.
6. (опційно) **Прикінцева цитата `>`** — зв'язок з дорожньою картою або з
   іншими боргами. Якщо в проєкті немає на що посилатись — пропустити.

Повний шаблон з прикладом — у `references/item-template.md`. Зчитати перед
створенням першого item або коли треба перевірити порядок блоків.

## Workflow: створення нового боргу

### 1. Перевірити наявність каталогу `tech_debt/`

- Якщо `tech_debt/index.md` **існує** — зчитай його, щоб дізнатись найбільший
  використаний номер і перевірити, чи такий борг уже не зафіксовано.
- Якщо `tech_debt/` **відсутній** — створи його разом з порожнім скелетом
  `index.md` за шаблоном з `references/index-template.md`. Користувача
  попередь: «У репо ще немає `tech_debt/`, створюю каталог разом з реєстром».

### 2. Перевірити CLAUDE.md проєкту

Подивись, чи `CLAUDE.md` у корені репо вже містить маркер
`<!-- tech-debt-skill -->`. Якщо ні:

- Прочитай готовий блок з `references/claude-md-snippet.md`.
- Покажи його користувачеві й попроси дозволу додати у `CLAUDE.md` (якщо
  `CLAUDE.md` відсутній — запропонувати створити з цим блоком).
- Без явного «так» — не редагувати `CLAUDE.md`. Якщо відмовив чи проігнорував
  — продовжуй створення боргу без CLAUDE.md.

### 3. Оцінити критичність і часовий горизонт

Перш ніж писати item, визнач:

- **Критичність** (`🔴` / `🟡` / `🟢`): чи блокує, чи лише впливає на DX/UX, чи
  приємно мати.
- **Часовий горизонт**: чи цей борг має братись у роботу **зараз**, чи це ідея
  на далеке майбутнє з умовою активації.

Сигнали на відкладений борг:

- користувач явно каже «це після релізу X», «це коли впровадимо Y»
- «не зараз, нагадай у Q3»
- «коли матимемо платних користувачів»
- «після переходу на нову БД»

**Якщо з контексту неясно — запитай**: «Цей борг для роботи зараз, чи
відкладений до конкретної умови/дати? Якщо відкладений — після чого його
брати?»

Без явної відповіді не виставляй статус `🕒 відкладено` самовільно. За
замовчуванням — `відкрито`.

### 4. Зібрати дані для item

Запитай користувача (одним повідомленням, не по одному):

- **Коротка назва** (для заголовка та slug)
- **Категорія** (вільне поле)
- **Пріоритет** (🔴 / 🟡 / 🟢)
- **Статус** і, якщо `🕒 відкладено`, — **умова активації**
- **Опис проблеми** (контекст, чому болить)
- **Варіанти рішення** (хоча б один; ідеально 2-3 з плюсами/мінусами)
- **Попередня рекомендація** (один з варіантів + обґрунтування)
- **Критерії «зроблено»** (чек-лист)
- **Дотичні файли** (опційно, якщо вже відомі)

Якщо контекст уже відомий з попередньої розмови — не перепитуй, а покажи
драфт і попроси доповнити чи затвердити.

### 5. Згенерувати slug і номер

- **Номер**: `(макс у index.md) + 1`, завжди трицифровий. Не використовувати
  «дірки» від видалених/закритих боргів.
- **Slug**: kebab-case з короткої назви (українські букви транслітерувати або
  замінити англійською коротко-описовою фразою; домовляйся з користувачем,
  якщо неоднозначно).

### 6. Створити item-файл

Шлях: `tech_debt/items/NNN-slug.md`. Структура — за шаблоном з
`references/item-template.md`. Усі шість блоків у порядку (шостий —
опційний). Поле `Активувати після:` — лише для відкладених.

### 7. Додати рядок у реєстр

У таблиці `tech_debt/index.md` додай рядок:

```
| NNN | [Коротка назва](items/NNN-slug.md) | Категорія | 🟡 середній | відкрито |
```

Або для відкладеного:

```
| NNN | [Коротка назва](items/NNN-slug.md) | Категорія | 🟢 низький | 🕒 відкладено |
```

Зберігай порядок номерів зростаючим.

### 8. Показати diff і нагадати про коміт

- Покажи створені/змінені файли.
- Нагадай: **«Відкриття тех боргу — окрема задача від основної. Закомить ці
  зміни окремим комітом, не змішуй з робочою задачею.»**
- **НЕ** запускай `git add`, `git commit`, `git push` автоматично.

## Workflow: редагування наявного боргу

Редагування — це зміна метаданих, опису, варіантів, рекомендації чи критеріїв
**без** видалення запису.

### 1. Знайти борг

Знайди `tech_debt/items/NNN-slug.md` за номером, який вказав користувач. Якщо
файла немає — покажи список відкритих боргів з `index.md` і попроси уточнити.

### 2. Визначити, що саме редагується

Можливі типи правок:

- **Пріоритет** (🔴 ↔ 🟡 ↔ 🟢) — оновити поле `Пріоритет:` в item і колонку
  `Пріоритет` у реєстрі.
- **Статус** (`відкрито` ↔ `🕒 відкладено`) — оновити поле `Статус:` в item і
  колонку `Статус` у реєстрі.
  - Перехід у `🕒 відкладено`: **додати** поле `Активувати після:` з умовою.
  - Перехід з `🕒 відкладено` у `відкрито`: **прибрати** поле `Активувати
    після:`.
- **Категорія** — оновити поле `Категорія:` в item і колонку у реєстрі.
- **Назва** — оновити заголовок item і текст посилання у реєстрі. **Номер і
  ім'я файлу (slug) не змінюються** — щоб не ламати посилань.
- **Проблема / Варіанти / Рекомендація / Критерії / Дотичні файли** —
  оновлювати тільки в item-файлі. Реєстр чіпати не треба.

### 3. Зробити правки точково

- Відкрий потрібний(і) файл(и).
- Зміни лише ті блоки, які стосуються запиту.
- Не переформатовуй item-файл цілком; зберігай порядок шести блоків і стиль
  існуючого вмісту.

### 4. Перевірити узгодженість

Після правок переконайся:

- Метадані в item і відповідний рядок у реєстрі збігаються (`Пріоритет`,
  `Статус`, `Категорія`, посилання на файл).
- Якщо `Статус: 🕒 відкладено` — поле `Активувати після:` присутнє і
  непорожнє.
- Якщо `Статус: відкрито` — поля `Активувати після:` немає.

### 5. Нагадати про коміт

- Покажи diff.
- Рекомендуй: редагування боргу — це теж окрема задача від основної робочої
  задачі; зробити окремим комітом.
- **НЕ** запускай `git` автоматично.

## Чого скіл НЕ робить

- **Не виконує** (не закриває) борг — це `tech-debt-close`. Якщо користувач
  каже «зроби борг NNN» / «закрий борг NNN» — підкажи активувати
  `tech-debt-close`.
- **Не видаляє** борг як неактуальний — це теж `tech-debt-close`.
- **Не комітить** і не пушить нічого. Тільки створює/редагує файли в
  `tech_debt/`.
- **Не вирішує сам** між варіантами A/B/C — це продуктове рішення
  користувача. Можна давати рекомендацію, але остаточно обирає користувач.
- **Не змінює номер чи slug** наявного боргу при редагуванні назви — щоб не
  ламати посилання.
- **Не виставляє `🕒 відкладено` без явної умови активації** — поле
  `Активувати після:` обов'язкове для цього статусу.
- **Не редагує CLAUDE.md** без явного підтвердження.
- **Не модифікує робочий код** у відповідь на «запиши в техборг» — якщо
  користувач каже «це поки заглушка, запиши в техборг», ти ТІЛЬКИ записуєш,
  не фіксиш самі.
- **Не створює дублікат, якщо борг уже є**. Якщо схожа назва вже у реєстрі —
  спершу запитай, чи це не той самий борг.

## Якщо борг уже існує (при створенні)

Перед створенням пробіжись по назвах у `index.md`. Якщо схожа назва вже є —
покажи користувачеві наявний item і запитай: «Це той самий борг? Якщо так,
може, оновити існуючий через редагування замість створювати дублікат?»

## Якщо номер не знайдено (при редагуванні)

Якщо користувач посилається на `#NNN`, якого немає в `index.md`:

- Можливо, борг уже закрито/видалено раніше (перевір git log).
- Покажи поточний список відкритих боргів і попроси уточнити номер.
- Не створюй автоматично новий борг під цим номером — номери не
  перевикористовуються.
