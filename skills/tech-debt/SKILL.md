# Tech Debt — управління технічним боргом

Скіл керує технічним боргом у репозиторії проєкту за усталеною структурою OneB:
реєстр-таблиця + окремий файл на кожен борг. Покриває три сценарії:

1. **Реєстрація нового боргу** (`workflow створення нового боргу` нижче)
2. **Перевірка зв'язку поточної задачі з боргом** (`перевірка дотичності` нижче)
3. **Проактивна поведінка** під час будь-якої задачі з кодом (`проактивний режим` нижче)

## Коли активуватись

**Реєстрація — явні тригери, діяти одразу:**

- «запиши в техборг» / «додай в техборг» / «запиши тех борг» / «відкрий тех борг»
- «новий тех борг» / «новий tech debt»
- «record tech debt» / «log tech debt» / «open tech debt item» / «tech debt item»
- Згадка `tech_debt/` у контексті запису

**Реєстрація — неявні сигнали, спершу запропонувати:**

- «в майбутньому треба буде зробити...»
- «потім ми будемо робити...»
- «це поки заглушка»
- «треба буде колись поправити»
- «зараз пропустимо, але повернемось»
- «це не наш кейс зараз, але...»

При неявному сигналі: коротко запитати в користувача «Зафіксувати це як тех борг
у `tech_debt/`?» і чекати «так». Без явного підтвердження запис не створювати.

**Перевірка дотичності та закриття/розширення — фрази:**

- «це пов'язано з тех боргом?» / «глянь, чи це не закриває борг»
- «це часом не item NNN?» / «можемо заодно закрити борг?»
- «це перекриває якийсь з відкритих боргів?»

**Проактивний режим — внутрішній тригер моделі:**

Активується сам, коли під час виконання задачі модель помічає, що змінюваний
код перетинається з областю відкритого боргу. Деталі — нижче в секції
«Проактивний режим».

## Структура каталогу

```
tech_debt/
├── index.md                       # реєстр-таблиця всіх відкритих боргів
└── items/
    ├── 001-kebab-case-slug.md
    ├── 002-another-slug.md
    └── ...
```

Іменування файлів: `NNN-kebab-case-slug.md`, де `NNN` — трицифровий номер
наступний за порядком у реєстрі (`001`, `002`, ..., `012`, ...).

## Реєстр (tech_debt/index.md)

Таблиця з колонками: `#`, `Назва (з посиланням)`, `Категорія`, `Пріоритет`,
`Статус`.

Пріоритети — три рівні з кольоровими маркерами:

- `🔴 високий` — блокує або спричиняє баги
- `🟡 середній` — впливає на DX/UX
- `🟢 низький` — приємно мати

Статус для відкритих — `відкрито`. Закритих у реєстрі не зберігаємо
(історія в git).

Точний шаблон реєстру — у `references/index-template.md`. Зчитати, коли
треба створити `index.md` з нуля або переконатись у форматі рядка.

## Структура item-файлу

Шість блоків у фіксованому порядку:

1. **Заголовок** — `# NNN — Коротка назва`
2. **Метадані** (буліст):
   - `Категорія:` — вільне поле (приклади: `Backend / Domain model`,
     `DX / Build`, `Observability / Operations`, `Frontend / State`,
     `Testing`, `Security`, `Performance`)
   - `Пріоритет:` — з емодзі (`🔴 високий` / `🟡 середній` / `🟢 низький`)
   - `Статус:` — `відкрито`
   - `Дотичні файли:` — список markdown-посилань на конкретні файли
     (`[path/to/file.ts](path/to/file.ts)`) + коротке пояснення поряд
3. **`## Проблема`** — контекст: що саме не так, чому це болить, які наслідки
   видно зараз. Може містити кодові блоки, цитати з логів.
4. **`## Куди рухатись`** — варіанти рішення (`Варіант A`, `Варіант B`,
   `Варіант C` — мінімум один) з плюсами/мінусами, опційно блоки коду.
   У кінці — `Попередня рекомендація:` з обґрунтуванням.
5. **`## Критерії "зроблено"`** — чек-лист `- [ ]` з конкретними
   перевірюваними умовами.
6. (опційно) **Прикінцева цитата `>`** — зв'язок з дорожньою картою або
   з іншими боргами. Якщо в проєкті немає на що посилатись — пропустити.

Повний шаблон з прикладом — у `references/item-template.md`. Зчитати перед
створенням першого item або коли треба перевірити порядок блоків.

## Workflow створення нового боргу

Виконати кроки в такому порядку:

### 1. Перевірити наявність каталогу `tech_debt/`

Подивись у корені репо:

- Якщо `tech_debt/index.md` **існує** — зчитай його, щоб дізнатись наступний
  вільний номер і перевірити, чи такий борг уже не зафіксовано.
- Якщо `tech_debt/` **відсутній** — створи його разом з порожнім скелетом
  `index.md` за шаблоном з `references/index-template.md`. Користувача
  попередь: «У репо ще немає `tech_debt/`, створюю каталог разом з реєстром».

### 2. Перевірити CLAUDE.md проєкту

Подивись, чи `CLAUDE.md` у корені репо вже містить маркер
`<!-- tech-debt-skill -->`. Якщо ні:

- Прочитай готовий блок з `references/claude-md-snippet.md`.
- Покажи його користувачеві й попроси дозволу додати у `CLAUDE.md`
  (якщо `CLAUDE.md` відсутній — запропонувати створити з цим блоком).
- Без явного «так» — не редагувати `CLAUDE.md`. Якщо користувач відмовив
  чи проігнорував — продовжуй створення боргу без CLAUDE.md.

### 3. Зібрати дані для item

Запитай користувача (одним повідомленням, не по одному):

- **Коротка назва** (для заголовка та slug)
- **Категорія** (вільне поле)
- **Пріоритет** (🔴 / 🟡 / 🟢)
- **Опис проблеми** (контекст, чому болить)
- **Варіанти рішення** (хоча б один; ідеально 2-3 з плюсами/мінусами)
- **Попередня рекомендація** (один з варіантів + обґрунтування)
- **Критерії «зроблено»** (чек-лист)
- **Дотичні файли** (опційно, якщо вже відомі)

Якщо контекст уже відомий з попередньої розмови — не перепитуй, а покажи
драфт і попроси доповнити чи затвердити.

### 4. Згенерувати slug і номер

- **Номер**: наступний за порядком у `index.md` (`001`, `002`, ...), завжди
  трицифровий.
- **Slug**: kebab-case з короткої назви (українські букви транслітерувати або
  замінити англійською коротко-описовою фразою; домовляйся з користувачем,
  якщо неоднозначно).

### 5. Створити item-файл

Шлях: `tech_debt/items/NNN-slug.md`. Структура — за шаблоном з
`references/item-template.md`. Усі шість блоків у порядку (шостий —
опційний).

### 6. Додати рядок у реєстр

У таблиці `tech_debt/index.md` додай рядок:

```
| NNN | [Коротка назва](items/NNN-slug.md) | Категорія | 🟡 середній | відкрито |
```

Зберігай порядок номерів зростаючим.

### 7. Показати diff і нагадати про коміт

- Покажи створені/змінені файли (через перегляд або pres ent_files).
- Нагадай: **«Відкриття тех боргу — окрема задача від основної. Закомить
  ці зміни окремим комітом, не змішуй з робочою задачею.»**
- **НЕ** запускай `git add`, `git commit`, `git push` автоматично.

## Чого скіл НЕ робить

- **Не комітить** і не пушить нічого. Тільки створює/редагує файли.
- **Не вирішує сам** між варіантами A/B/C — це продуктове рішення
  користувача. Можна давати рекомендацію, але остаточно обирає користувач.
- **Не закриває** борги. Це робота окремого workflow (видалення файлу +
  рядка з реєстру виконує користувач або інший майбутній скіл).
- **Не редагує CLAUDE.md** без явного підтвердження.
- **Не модифікує** робочий код у відповідь на «запиши в техборг» — якщо
  користувач каже «це поки заглушка, запиши в техборг», ти ТІЛЬКИ записуєш,
  не фіксиш самі.

## Якщо борг уже існує

Перед створенням пробіжись по назвах у `index.md`. Якщо схожа назва вже є —
покажи користувачеві наявний item і запитай: «Це той самий борг? Якщо так,
може, оновити існуючий замість створювати дублікат?»

## Проактивний режим

Цей режим — про поведінку **під час будь-якої задачі з кодом**, навіть коли
користувач не казав «тех борг». Логіка:

### На початку нетривіальної задачі

Якщо у репо є `tech_debt/index.md` — швидко глянь у нього і запам'ятай назви
відкритих боргів. Якщо назви/категорії перетинаються з тим, що ти збираєшся
змінювати, прочитай відповідний `tech_debt/items/NNN-*.md` (особливо секції
«Куди рухатись» з рекомендованим варіантом і «Критерії "зроблено"»).

Не треба читати всі items підряд — тільки ті, що схоже стосуються поточної
області.

### Три правила під час виконання

1. **Не збільшуй борг.** Не додавай нового коду, який поглиблює проблему,
   описану в дотичному боргу. Приклад: якщо item каже «треба піти від
   глобальних змінних», не плоди нові глобальні змінні.

2. **Зменшуй, коли це безкоштовно.** Якщо у новому коді природно лягає
   «новий підхід» з рекомендованого варіанту боргу, **і** це не ламає
   поточну логіку, **і** не додає значної додаткової роботи — користуйся
   ним. Це може автоматично закрити частину критеріїв «зроблено» у боргу.

3. **Сигналь явно, коли вибрав «старий підхід».** Якщо все-таки використав
   застарілий патерн (бо «новий» зламав би логіку чи коштував дорого) —
   коротко скажи це в кінці відповіді. Один рядок, без драматизації:
   «Тут лишив старий патерн X, бо новий вимагав би переписати Y — це не
   ускладнило ситуацію з item NNN, але й не просунуло.»

### Закриття або розширення задачі

Якщо помічаєш, що поточна задача **дуже близька** до відкритого боргу:

- **Якщо закриття боргу робить поточну задачу зайвою** (item розв'язує саме
  той сценарій, який зараз просять зробити «локально») → запропонуй
  користувачу:

  > «Глянь, item NNN, схоже, перекриває цю задачу. Замість локального фіксу
  > може краще закрити борг — це дасть структурне рішення. Зробимо так?»

- **Якщо невелике розширення поточної задачі закриває борг або суттєво його
  просуває** (виконує більшість критеріїв «зроблено») → запропонуй:

  > «За приблизно +X хвилин роботи поверх поточної задачі можна заодно
  > закрити item NNN. Робимо?»

**В обох випадках завжди питай користувача**, не вирішуй сам. Без явного
«так» — продовжуй поточну задачу як було.

### Чого НЕ робити у проактивному режимі

- **Не зупиняй поточну задачу**, щоб «перевірити всі борги». Звіряйся
  тільки з тими, що дотичні до області змін.
- **Не переписуй чужий код «по дорозі»**, тільки тому що item каже «треба
  піти від X». Те, чого item стосується, лишається окремою задачею
  (видалення/оновлення item — це окремий коміт від робочої задачі).
- **Не закривай item** автоматично, навіть якщо всі критерії стали
  виконані — це робить користувач (або майбутній скіл `tech-debt-close`).
