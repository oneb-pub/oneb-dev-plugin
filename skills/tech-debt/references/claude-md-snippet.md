# CLAUDE.md snippet

Готовий блок, який скіл `tech-debt` пропонує додати у `CLAUDE.md` проєкту
при першій активації, якщо у файлі ще немає маркера `<!-- tech-debt-skill -->`.

Цей блок робить дві речі:

1. Інформує Claude, що в проєкті є структурований реєстр тех боргу — щоб у
   відповідях не вигадувати власних форматів.
2. Прайминг на неявні сигнали («в майбутньому треба буде...», «потім будемо
   робити...») — щоб Claude пропонував зафіксувати їх як борг, навіть без
   явного «запиши в техборг».

## Блок для вставки

````markdown
<!-- tech-debt-skill -->
## Технічний борг

Реєстр тех боргу живе в `tech_debt/`:

- `tech_debt/index.md` — таблиця всіх відкритих боргів (`#`, назва-посилання, категорія, пріоритет, статус)
- `tech_debt/items/NNN-slug.md` — окремий файл на кожен борг з повним описом (проблема, варіанти рішення, критерії «зроблено»)

Закриті борги фізично видаляються з реєстру та `items/` — історія залишається в git.

### Реєстрація нового боргу

Коли користувач каже одне з:

- «запиши в техборг» / «додай в техборг» / «новий тех борг»
- «в майбутньому треба буде зробити...»
- «потім ми будемо робити...»
- «це поки заглушка»
- «треба буде колись поправити»

→ використай скіл `tech-debt`, щоб коректно зафіксувати запис.

При неявних сигналах (другий блок вище) — спершу запитай: «Зафіксувати це як тех борг?». Без явного «так» — не створюй запис.

### Проактивна поведінка під час будь-якої задачі з кодом

На початку нетривіальної задачі — швидко глянь у `tech_debt/index.md`. Якщо там є борги, дотичні до файлів/областей, які ти збираєшся змінювати — прочитай відповідний `items/NNN-*.md` (особливо секцію «Куди рухатись» з рекомендованим варіантом).

Далі під час виконання дотримуйся **трьох правил**:

1. **Не збільшуй борг.** Не додавай нового коду, який поглиблює проблему, описану в боргу (наприклад, не плоди ще одне використання застарілої API, якщо борг саме про те, щоб від неї піти).

2. **Зменшуй, коли це безкоштовно.** Якщо у новому коді природно лягає «новий підхід» з рекомендованого варіанту боргу, **і** це не ламає поточну логіку, **і** не додає значної додаткової роботи — користуйся ним. Мета: новий код не лише не псує борг, а ще й трохи його закриває (наприклад, кілька критеріїв «зроблено» відразу виконуються).

3. **Сигналь явно, коли вибрав «старий підхід».** Якщо в новому коді все-таки використав застарілий патерн (бо «новий» зламав би логіку чи коштував дорого) — коротко скажи це в кінці відповіді. Без драматизації, просто факт.

### Закриття або розширення задачі

Якщо помічаєш, що поточна задача **дуже близька** до відкритого боргу:

- **Якщо закриття боргу робить поточну задачу зайвою** (тобто борг розв'язує саме той сценарій, який зараз просять зробити «локально») → запропонуй користувачу: «Глянь, item NNN, схоже, перекриває цю задачу. Може, краще закрити борг замість локального фіксу?»

- **Якщо невелике розширення поточної задачі закриває борг або суттєво просуває його вперед** (виконує більшість критеріїв «зроблено») → запропонуй: «За +X хвилин роботи можна заодно закрити item NNN. Робимо?»

В обох випадках **завжди питай користувача**, не вирішуй сам. Без явного «так» — продовжуй поточну задачу як було.

### Правила, спільні для всього

- Відкриття/закриття тех боргу — **окрема задача** від основної. Не змішуй ці зміни у комітах з робочою задачею.
- Скіл `tech-debt` сам не комітить і не пушить — це робиш ти або користувач.
- Не вирішуй сам між варіантами рішення (A/B/C) у боргу — це продуктове рішення.
- Якщо `tech_debt/` у проєкті немає — пропусти проактивну частину (нема з чим звіряти). Реєстрація нового боргу однак працює — скіл створить структуру.
<!-- /tech-debt-skill -->
````

## Інструкція для скіла

При використанні цього snippet:

1. Зчитай `CLAUDE.md` у корені поточного репо.
2. Якщо `CLAUDE.md` не існує — запропонуй створити з цим блоком (та запитом
   на підтвердження).
3. Якщо `CLAUDE.md` існує і вже містить `<!-- tech-debt-skill -->` — нічого
   не роби.
4. Якщо `CLAUDE.md` існує без маркера — покажи блок користувачеві й попроси
   дозволу додати у кінець файлу (зі збереженням існуючого вмісту).

**Не модифікуй `CLAUDE.md` мовчки. Завжди — явне підтвердження.**
