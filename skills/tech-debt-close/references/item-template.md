# Шаблон item-файлу

Структура файлу `tech_debt/items/NNN-slug.md` — шість блоків у фіксованому
порядку. Шостий блок (прикінцева цитата) опційний. Метаблок містить
обов'язкові поля (`Категорія`, `Пріоритет`, `Статус`, `Дотичні файли`) і
одне опційне поле (`Активувати після`), яке стає обов'язковим за умови
`Статус: 🕒 відкладено`.

## Порожній шаблон (відкладений борг)

Відрізняється від відкритого лише метаблоком — у нього додано поле
`Активувати після:` і змінено `Статус:`:

```markdown
# NNN — Коротка назва боргу

- **Категорія:** Performance
- **Пріоритет:** 🟢 низький
- **Статус:** 🕒 відкладено
- **Активувати після:** після релізу v2.0 (запуск нового API)
- **Дотичні файли:**
  - [path/to/file.ts](path/to/file.ts) — пояснення

## Проблема
...
```

Значення `Активувати після:` — одне з:

- конкретна дата у форматі `YYYY-MM-DD`: `2026-09-01`
- реліз або мілстоун: `після релізу v2.0`, `після виходу з беті`
- умова з кодової бази: `після впровадження auth-модуля`, `після переходу
  на нову БД`
- сценарне обмеження: `коли матимемо платних користувачів`, `при перевищенні
  10k RPS`

## Приклад заповненого item-файлу (відкритий)

```markdown
# 002 — Налаштувати Sentry для бекенду

- **Категорія:** Observability / Operations
- **Пріоритет:** 🔴 високий
- **Статус:** відкрито
- **Дотичні файли:**
  - [config/sentry.ts](config/sentry.ts) — наразі порожній stub
  - [src/server.ts](src/server.ts) — тут треба зачепити middleware

## Проблема

Зараз помилки на проді ловляться тільки через console.error → docker logs.
Це не масштабується: коли впала фонова задача, ми дізнаємось про це з
запізненням і без stack trace. На staging уже двічі за тиждень була
прихована регресія.

## Куди рухатись

### Варіант A — Sentry SaaS

Підключити стандартний Sentry SDK.

- **Плюси:** швидко, готовий dashboard, alerts, sourcemaps з коробки.
- **Мінуси:** коштує гроші, дані йдуть назовні.

### Варіант B — self-hosted GlitchTip

GlitchTip — open-source клон Sentry, сумісний по API.

- **Плюси:** дані в нас, безкоштовно.
- **Мінуси:** треба тримати інфру, менше features.

**Попередня рекомендація:** Варіант A. На нашому масштабі собівартість часу
на self-host > вартість Sentry. Якщо виростемо — переїдемо.

## Критерії «зроблено»

- [ ] Sentry SDK ініціалізується при старті бекенду
- [ ] DSN читається зі змінної оточення (інструкція для користувача в .env.example)
- [ ] Усі необроблені exceptions летять у Sentry
- [ ] Source maps підвантажуються при деплої
- [ ] Помилка з тестового endpoint видно у дашборді

> Пов'язано з Phase 0.0 — Observability foundation.
```

## Приклад заповненого item-файлу (відкладений)

```markdown
# 014 — Перейти з REST на GraphQL для публічного API

- **Категорія:** Backend / API
- **Пріоритет:** 🟢 низький
- **Статус:** 🕒 відкладено
- **Активувати після:** після релізу v2.0 і досягнення 1k активних
  інтеграцій (раніше витрати на міграцію не окупляться)
- **Дотичні файли:**
  - [src/api/routes/](src/api/routes/) — поточні REST-роути
  - [docs/api.md](docs/api.md) — публічна документація

## Проблема

Партнери просять складніших запитів з агрегацією — на REST доводиться
ліпити багато endpoint-ів-варіантів. Це поки терпимо, але після виходу
v2.0 трафік зросте і кожен новий партнерський use-case плодитиме нові
REST-роути.

## Куди рухатись

### Варіант A — Apollo Server поверх існуючих сервісів

Підняти GraphQL-шар як фасад над поточними сервісами.

- **Плюси:** не ламає REST, можна мігрувати поступово.
- **Мінуси:** дві паралельні поверхні API на проді.

### Варіант B — Замінити REST на GraphQL повністю

Жорстка міграція з версіонуванням.

- **Плюси:** одна точка входу, чистий контракт.
- **Мінуси:** ламає всіх існуючих клієнтів, потребує комунікації.

**Попередня рекомендація:** Варіант A на старті, з планом перейти у B
через рік після стабілізації схеми.

## Критерії «зроблено»

- [ ] GraphQL endpoint піднято і доступний партнерам
- [ ] Схема покриває топ-5 партнерських use-case
- [ ] Документація з прикладами queries / mutations
- [ ] Перевірено навантаженням ≥ поточного REST
- [ ] Мінімум один партнер мігрував і дав фідбек

> Активувати після підтвердження виконання умови (див. метаблок).
```

## Контракти писемності

- **Заголовок item**: `# NNN — Коротка назва` (em-dash, не дефіс).
- **Метадані**: чотири обов'язкові пункти у фіксованому порядку
  (`Категорія`, `Пріоритет`, `Статус`, `Дотичні файли`) і одне опційне
  поле (`Активувати після`), яке вставляється **між** `Статус` і
  `Дотичні файли`, і присутнє **тоді і тільки тоді**, коли
  `Статус: 🕒 відкладено`.
- **Емодзі пріоритетів**: тільки `🔴`, `🟡`, `🟢`. Не використовувати інші.
- **Емодзі статусу**: `🕒` лише для значення `відкладено`. Для `відкрито` —
  без емодзі.
- **`Активувати після:`**: одне з допустимих форматів значень (див. вище).
  Без конкретного значення поле не існує. Якщо умова стала розмитою —
  переформулювати точніше або повернути борг у статус `відкрито`.
- **Варіанти рішення**: мінімум один, ідеально 2-3. Кожен варіант — це
  заголовок третього рівня `### Варіант X — назва`.
- **`Попередня рекомендація`** — обов'язково, навіть якщо варіант один
  (тоді обґрунтуй, чому він єдиний).
- **Чек-лист**: тільки markdown checkboxes `- [ ]`. Не використовувати
  emoji галочки.
- **Прикінцева цитата**: опційна. Якщо нема на що посилатись — пропусти,
  не залишай порожній блок.

## Правила змін у файлі

- При **редагуванні** — змінюй лише потрібні блоки, зберігай порядок і
  стиль решти. Не переформатовуй весь файл.
- При **перенесенні у відкладені** — додай поле `Активувати після:` між
  `Статус:` і `Дотичні файли:`; не міняй інших полів без потреби.
- При **знятті відкладеності** — повністю прибери поле `Активувати після:`
  (а не залишай порожнє); зміни `Статус:` на `відкрито`.
- При **зміні назви** — оновлюй тільки заголовок `# NNN — ...`. Номер і
  ім'я файлу (slug) не змінюються.
- При **виконанні** — після підтвердження виконання всіх критеріїв файл
  фізично видаляється (а не позначається `закрито`).
- При **видаленні як неактуального** — файл також фізично видаляється;
  історія залишається у git.
